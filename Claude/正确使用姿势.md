## 1. 接手一个新项目

把这段直接发给 Claude Code：

```text
请按 everything-claude-code 的方式接手这个项目：

1. 先阅读项目目录结构和关键配置文件
2. 识别技术栈、包管理器、运行方式、测试方式
3. 告诉我本地启动步骤
4. 标出最关键的入口文件和核心模块
5. 列出我第一次上手最该看的 5 个文件
6. 最后给我一个“10 分钟快速熟悉项目”的行动清单
```

这相当于手动复刻它的 **plan + project understanding** 工作流。ECC 的仓库本身就是按 `rules / commands / skills / agents` 组织的一套工程化流程。

---

## 2. 学习项目配置环境

这就是你现在最适合用的一条：

```text
请按 everything-claude-code 的工作流帮我学习这个项目的配置环境：

1. 阅读 package.json / pyproject.toml / go.mod / Dockerfile / README / CI 配置
2. 总结运行环境要求、依赖安装方式、环境变量、数据库或外部服务要求
3. 给我一份从零启动的步骤
4. 列出常见启动失败原因和排查方法
5. 最后给我一个最小可运行验证清单
```

---

## 3. 开发一个新功能

ECC 的核心思路就是 **先规划，再实现，再验证**。README 里列出的核心命令也围绕 `plan / tdd / code-review / verify` 这一类流程。

你可以直接这样问：

```text
我要做一个“用户登录”功能，请按 everything-claude-code 风格执行：

1. 先给我实现方案
2. 拆成最小可提交步骤
3. 先列测试点
4. 再开始改代码
5. 每改完一部分就告诉我如何验证
6. 最后做一次代码审查和风险检查
```

---

## 4. 排查环境问题 / 启动失败

这个很适合你现在的场景：

```text
请按 everything-claude-code 风格帮我排查项目无法启动的问题：

1. 先识别启动命令和依赖前置条件
2. 检查环境变量、依赖安装、版本要求、端口占用、数据库连接
3. 给我一个最可能原因优先级列表
4. 每次只让我执行一个最有价值的排查步骤
5. 最后总结根因和修复方案
```

---

## 5. 写测试

虽然你机器上 `/tdd` 命令没稳定跑起来，但 ECC 仓库里确实把 TDD 作为正式命令和工作流来提供。  
所以你直接用自然语言照着它的思路来就行：

```text
请按 everything-claude-code 的 TDD 工作流为这个功能补测试：

1. 先总结要测什么
2. 先写失败测试
3. 再写最小实现
4. 再补边界条件
5. 最后告诉我如何运行这些测试
```

---

## 6. 做代码审查

```text
请按 everything-claude-code 的 code review 风格审查我当前改动：

1. 检查逻辑正确性
2. 检查边界条件和异常处理
3. 检查可读性和可维护性
4. 检查安全问题
5. 检查测试是否足够
6. 按“必须改 / 建议改 / 可选优化”输出
```

ECC 的规则层里本来就强调编码风格、测试、安全和工作流约束，所以这种问法最贴它的用法。

---

## 7. 做最终验证

```text
请按 everything-claude-code 的 verify 思路帮我做交付前检查：

1. 功能是否满足需求
2. 测试是否覆盖关键路径
3. 是否有明显回归风险
4. 是否缺少文档或配置说明
5. 给我一个发布前检查清单
```

---

## 8. 你最常用的一套连招

以后你接一个任务，直接按这个顺序聊：

```text
先帮我理解项目和需求
→ 再给实现计划
→ 再列测试点
→ 再改代码
→ 再做 code review
→ 再做 verify
```

这就是 ECC 最核心的价值：  
**把 AI 写代码，从“直接开写”变成“规划—实现—测试—审查—验证”的闭环。** 仓库的命令和目录设计就是围绕这个闭环展开的。
