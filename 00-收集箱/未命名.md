本次测试全过程纪要

  1. 本次任务最初目标是“根据《Java组件使用（一）》内训视频，整理内训笔记，运行示例代码并在评论区提交笔记和运行截图”。一
     开始明确要求是不直接改业务代码，只给操作步骤。
  2. 首先确认了本次训练主要涉及的示例代码位置，重点包括 eams-sample 模块中的 Excel、FastDFS、示例控制器与测试类，以及公
     共组件扫描配置，核心文件包括 UseCusComponents.java、ExcelController.java、FileController.java、ExcelTest.java。
  3. 第一次尝试启动 eams-sample 时，项目报错：
     Could not resolve placeholder 'sp.sample' in value "${sp.sample}"。这说明不是业务代码先出问题，而是应用启动时缺少端
     口和服务名配置，问题定位到 eams-sample 的 application.yaml 中 server.port=${sp.sample} 和
     spring.application.name=${sn.sample} 这两个占位符。
  4. 随后检查了 Nacos 配置中心。确认当前命名空间中已有 system.yaml、data-source.yaml、third-service.yaml、key-
     config.yaml 等配置，但项目实际通过 bootstrap.properties 读取的是 third-services.yaml，所以发现了一个关键不匹配问
     题：Nacos 中配置名是 third-service.yaml，而代码要求的是 third-services.yaml。
  5. 在继续处理 sp.sample 问题的过程中，又出现了编译环境报错：
     java.lang.NoSuchFieldError: com.sun.tools.javac.tree.JCTree$JCImport.qualid。这一步判断为 JDK 与 Lombok/注解处理器
     不匹配导致的编译问题，说明还没有真正进入业务启动阶段。
  6. 随后排查本机 JDK 环境。发现本机存在多个 JDK 目录，但可用性不一致：旧的 JDK8 实际版本过老，JDK11 目录为空，可用的是
     JDK17 和 JDK21。为了保证项目最起码能编译，额外补了项目级 JDK17 运行方式和一个 Maven 辅助脚本，用于强制以 JDK17 运行
     构建命令。
  7. 继续回到启动问题后，发现 sp.sample 仍未解析。进一步查看完整日志后，定位到根因并不是“配置缺失”，而是 Nacos 中多份
     YAML 根本没有被正确加载，因为配置内容本身格式错误。日志明确出现了 parse data from Nacos error，并且 system.yaml、
     data-source.yaml、third-services.yaml 都报了 YAML 解析失败。
  8. 因此改为逐个修复 Nacos 配置文件格式。重点处理了 system.yaml、data-source.yaml 和 third-services.yaml 的缩进、换行、
     URL 被断开等问题，尤其修复了 sp.sample / sn.sample、MySQL URL、FastDFS、RocketMQ、Seata 等关键配置项。至此，eams-
     sample 至少能正确读取到外部配置。
  9. 在排查环境的过程中，还专门梳理了数据库脚本位置和导入顺序。确认数据库脚本目录在 数据库目录，并梳理出 6 个 SQL 文件，
     其中正式初始化需要 5 个：zo_eams.sql、zo_eams_init_dict.sql、zo_eams_init_region.sql、zo_eams_init_sys.sql、
     zo_eams_init_user.sql，zo_eams_test_data.sql 为可选测试数据脚本。
  10. 同时对“是否必须把所有数据库脚本都创建完”做了说明。结论是，如果只是先跑组件示例，并不需要一开始就把所有 SQL 全量导
     入；EasyExcel、FastDFS、WebSocket、RocketMQ 可以先按最小环境逐步验证，而涉及登录、OAuth2、Seata 时才必须补齐相应数
     据库结构。
  11. 之后正式规划了本次组件测试顺序。为了最大化出结果和截图效率，建议顺序为：EasyExcel、FastDFS、WebSocket、RocketMQ、
     声明式服务、Seata。后续实际测试基本也是按这个思路推进的。
  12. FastDFS 测试阶段首先遇到的是下载报错：
     getStoreStorage fail, errno code: 22。这一步先判断为下载接口参数 group / storageId 可能拆分错误，或者上传返回的 URL
     被整段传给了下载接口，而不是按 FastDFS 的 group 和存储路径拆开传递。
  13. 紧接着在上传和下载链路中又先后出现了 errno code: 2、Connection reset、浏览器访问返回 ERR_CONNECTION_REFUSED 等问
     题。此时排查重点从业务控制器转移到了 FastDFS 环境本身，包括 tracker、storage、nginx 映射口、Docker 端口映射和宿主机
     访问链路。
  14. 通过 Linux 侧端口排查发现 22122 和 23000 在监听，但 8888 没有监听。也就是说，FastDFS 的 tracker 和 storage 在，但
     对外 HTTP 文件访问层没有起来，所以浏览器直接访问上传返回的 URL 会被拒绝。这一步解释了为什么下载接口和浏览器直链都不
     稳定。
  15. 围绕 FastDFS 又继续检查了 tracker/storaged 的注册状态、容器端口映射、宿主机访问、Nacos 中 fastdfs.tracker-servers
     和 fastdfs.nginx-servers 配置，以及 storage 注册 IP 是否错误为容器内网 IP。最终在修正服务端链路后，文件上传、直链访
     问和下载流程被逐步打通。
  16. 在 FastDFS 下载成功后，又专门补充了“下载成功后还需要测试什么”的说明，明确还要测上传接口返回 URL、删除接口、删除后
     文件失效，以及异常参数场景。随后用户反馈“都测完了”，说明 FastDFS 这一组核心功能已经基本完成。
  17. 接下来进入 WebSocket 模块。初始尝试启动 eams-sample-ws 时，又出现了新的阻断问题：
     org.springframework.cloud.stream.binder.BinderException
     以及
     connect to [192.168.146.130:9876] failed。这里判断不是 WebSocket 本身坏了，而是 eams-sample-ws 启动时同时绑定了
     RocketMQ 消费者，所以 RocketMQ NameServer 不可用会直接影响 WebSocket 启动。
  18. 进一步检查 eams-sample-ws 配置，确认它在 application.yaml 中配置了 RocketMQ binder，并在 WsApplication.java 中使用
     了 @EnableBinding(Sink.class)，因此 RocketMQ 环境必须先正常。
  19. RocketMQ 方面一开始测试时 9876 和 10911 无法确认，后续在 Linux 侧执行 ss -lntp | grep 9876 和 ss -lntp | grep
     10911 后，最终确认 NameServer 和 Broker 都已经在监听。这意味着 RocketMQ 服务端本身存在，后续问题更多转移到客户端配
     置和模块联调上。
  20. WebSocket 页面接入时又遇到两个典型问题。第一，chat.html 中 WebSocket 地址写死成了
     ws://localhost:20003/chat，但实际 WsApplication 启动端口一度是 10800，导致 localhost 访问被拒绝。第二，chat.html
     文件放在 templates 目录而不是 static，又没有额外页面映射，所以直接访问 http://localhost:端口/chat.html 返回了 Whit
     elabel 404。
  21. 因此又针对 WebSocket 做了两类调整性操作建议：一类是通过 Nacos 把 sp.ws 调整到 20003，让服务端端口与页面脚本写死的
     端口一致；另一类是不再通过应用 URL 直接打开页面，而是直接用浏览器打开本地文件 chat.html 进行测试。
  22. 在真正建立 WebSocket 连接时，又发现它不是简单输入用户 ID，而是要求传入有效 JWT。进一步定位到
     WebSocketConfig.java，确认服务端会从 token 中解析用户 id 作为连接标识，因此登录链路必须先打通。
  23. 随后测试登录接口时，先出现：
     Load balancer does not have available server for client: OAUTH2
     说明 eams-login 模块虽然在，但其依赖的 OAuth2 服务 OAUTH2 没有在线实例，所以登录无法完成。由此又转入 eams-oauth2 的
     启动排障。
  24. 启动 eams-oauth2 的第一次报错是：
     Table 'zo_eams.menu' doesn't exist
     说明 OAuth2 初始化阶段要读取 menu 表，但当前数据库中并没有满足其要求的表结构。这一步进一步暴露出：OAuth2 示例依赖的
     并不是主业务库原样结构，而是它 README 中额外给出的一套认证专用表结构。
  25. 在继续启动 OAuth2 的过程中，又遇到了更隐蔽的问题：
     Cannot load keys from store: class path resource [jwt.jks]
     以及
     Invalid keystore format。最终定位到资源目录中的 jwt.jks 实际已经被转换成了 PKCS12 格式，而代码启动时仍按 JKS 读取。
     通过比对 jwt.jks、jwt.jks.old 和 jwt.pfx，最终恢复了可用的 JKS 文件。
  26. 修复 jwt.jks 后，又继续暴露出数据库结构不匹配的问题。登录时首先报：
     Unknown column 'username' in 'field list'
     说明 OAuth2 查询 user 表时需要字段 username，但主业务库 zo_eams.sql 中的 user 表字段实际是 name/mobile/password 一
     套结构，与 OAuth2 示例实体 User.java 并不一致。
  27. 因此围绕 OAuth2 又专门补了一套最小认证结构：在现有 user 表上增加 username 字段，插入 admin/test 两个测试用户，并补
     齐 role、user_role、menu、role_menu 等表及示例数据。中间又出现了：
     Table 'zo_eams.role' doesn't exist
     进一步验证了这套角色权限表确实必须单独补齐。
  28. 在这之后，又说明了一个测试误区：直接在浏览器地址栏访问 POST /oauth/token 或 POST /login/auth-login 只会得到 405
     Method Not Allowed，因为浏览器发的是 GET。于是转而改用 Knife4j、PowerShell 或 Postman 以 POST 方式去调用登录接口和
     OAuth2 原生接口。
  29. 随着 OAuth2、Login、WebSocket 三条链路逐步打通，接下来又梳理了 WebSocket 和 RocketMQ 联调的完整路径：先从登录接口
     拿 token，再在本地打开聊天页，将 token 原样贴入，不加 Bearer 前缀，然后用两个窗口分别连接两个用户，先测页面内点对点
     和 all 群发，再通过 eams-sample-rmq 的 POST /rmq/publish 用 clientId/message 参数向在线客户端推送消息。
  30. 聊天室跑通后，又总结了这一块的收尾动作，包括截图要求、评论区应该写的要点，以及接下来继续测试时的建议顺序。用户随后
     询问“跑完聊天室然后呢”，说明 WebSocket 与 RocketMQ 联调已进入可交付阶段。
  31. 最后进入 Seata 阶段。第一次调用 eams-sample 的 /sample/testSeata 时出现：
     Load balancer does not have available server for client: SEATA
     说明 eams-sample-seata 还没有正确注册到 Nacos，或者 sn.seata / sp.seata 没配好。
  32. 在启动 eams-sample-seata 之后，又出现了：
     远端保存失败
     查看代码后确认，这不是 Seata Server 本身直接报错，而是 eams-sample 本地服务先调远端 POST /seata/testSave，只要远端
     返回值不是 1 就会主动抛异常。对应调用在 ISeataFeignService.java 和 SampleServiceImpl.java。
  33. 进一步直接测试远端接口 POST /seata/testSave，第一次返回 0，说明远端执行了 SQL，但没有更新到任何数据行。继续追代码
     发现远端 TransServiceImpl.java 写死了一条 UPDATE sample SET name='我改了' WHERE
     id='00ed057321f944fb8568c9a8e647d632' 的 SQL，所以只有当 sample 表里存在这条固定 ID 数据时才会返回 1。
  34. 随后用户又反馈远端接口返回了 500。这说明问题从“没更新到数据”升级成了“命中数据后事务过程报错”。因为确认了 sample 表
     和数据都存在，所以排查重点又切到了 Seata AT 模式的 undo_log 表与事务提交过程。
  35. 在补齐相关数据和环境后，用户最终反馈“返回成功了”。这意味着 Seata 远端更新已经成功执行，本地通过 Feign 调远端并触发
     事务链的关键路径已经打通。
  36. 当所有这些模块逐步跑通后，又回过头总结了目前已完成的核心测试内容：EasyExcel 导入导出、FastDFS 上传下载删除、
     WebSocket 聊天、RocketMQ 消息推送、OAuth2/Login/Token 获取、Seata 分布式事务示例。剩余若继续扩展，可再补声明式服
     务 / Java 调 C++ 服务模块 eams-sample-cpp，但就本次作业交付来说，前述内容已经基本覆盖了训练重点。

  本次测试中涉及的主要问题类型汇总

  1. 配置中心问题
     包括 sp.sample 占位符缺失、Nacos Data ID 名称不匹配、YAML 格式错误导致配置加载失败。
  2. 编译与运行环境问题
     包括旧 JDK 版本、Lombok/注解处理器不匹配、IDEA 实际使用的 JDK 与预期不一致。
  3. 数据库问题
     包括主业务库表结构与 OAuth2 示例结构不一致、登录认证需要的 username/role/menu 等表缺失、Seata 远端测试数据缺失。
  4. 中间件与基础设施问题
     包括 FastDFS 的 8888 访问层未正常监听、RocketMQ 9876/10911 连通性、Seata 服务发现与事务依赖链。
  5. 页面与联调方式问题
     包括把 POST 接口当 GET 用浏览器打开导致 405、chat.html 作为模板文件不能直接通过 URL 访问、WebSocket 页面端口与后端
     端口不一致。

  本次测试最终结果

  6. eams-sample 基础模块最终可启动并用于组件验证。
  7. EasyExcel 导入导出流程已进入可测试和可截图状态。
  8. FastDFS 的上传、下载和访问链路经过环境修复后可用。
  9. WebSocket 页面、JWT 登录、RocketMQ 推送和聊天室联调已完成主要链路验证。
  10. Seata 示例最终返回成功，说明远端保存和事务示例路径已走通。
  11. 本次内训作业所需的“笔记整理 + 示例运行截图 + 问题排查过程”素材已基本齐备。
