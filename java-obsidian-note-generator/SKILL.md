# Java Obsidian 八股笔记生成器（增强版）

你被调用时，目标是：把 $ARGUMENTS 对应的 Java 知识点，生成一篇可直接放进 Obsidian 的“面试导向笔记”。

## 核心约束（必须遵守）

1) Obsidian 兼容
- 输出必须是纯 Markdown（可包含 Obsidian Callout：`> [!note]` 等）。
- 思维导图必须使用 Mermaid mindmap，并放在代码块中：
  - ```mermaid
    mindmap
      root((...))
    ```
- 不要输出不被 Obsidian/Markdown 支持的花哨语法。

2) 结构化覆盖（不许漏知识点）
无论主题大小，笔记必须包含以下固定模块（每个模块都要有内容，即使简短）：
- 定义与定位
- 应用场景
- 核心原理（分点 + 关键流程）
- 关键细节清单（面试高频点）
- 源码/实现要点（不写伪出处行号；写关键类/方法/机制）
- 易错点与陷阱
- 对比与扩展（至少 2 组对比）
- 标准面试回答（30 秒 / 2 分钟 / 深挖追问）
- 代码题与代码示例（必须“注释清晰”）
- 复习 Checklist（可勾选）
- Mermaid 思维导图（不求巨细，但要覆盖上面模块）

3) 内容质量
- 语言：中文为主，关键术语可中英对照（如 happens-before）。
- 风格：面试/考试导向，结论清晰，层次分明，避免空话。
- 专业性：不要胡编源码细节；如果不确定具体类/方法名，用“通常/一般位于…模块”并保持正确性。
- 深度控制：默认 interview（面试够用），如果参数包含 deep 再展开源码与边界。

4) 代码示例要求（强制）
- 必须给出至少 1 个可运行/可读的 Java 代码示例。
- 代码必须有逐段注释（解释意图、关键点、坑点）。
- 若是并发/线程安全主题：必须强调可见性/原子性/有序性或锁语义，并写明为什么安全/不安全。

## 输入解析（你自己做判断，不要反问用户）
从 $ARGUMENTS 中解析：
- topic：主题（如 HashMap 扩容、JVM 垃圾回收、Spring Bean 生命周期）
- level：interview / core / deep（默认 interview）
- tags：可选标签列表（用于 Obsidian frontmatter）

## 输出格式（严格按模板）
你必须使用 `template.md` 的结构生成整篇笔记。
- 若 topic 太大（如 JVM），也要保持模块齐全，但每模块只给“面试够用”的要点，不要写成教科书。
- 若 topic 太小（如 String 常量池），每模块也要补齐，允许简短。

## 参考资料（固定三站点）
在笔记末尾添加“参考与建议”时，固定为以下三条，名称保持不变：
- JavaGuide
- PDai
- 官方源码导读

不要贴大段链接；用简短条目说明“看什么、为啥”。

## 自检清单（输出前必须检查）
- [ ] 是否包含全部固定模块？
- [ ] Mermaid mindmap 是否能在 Obsidian 渲染（mindmap 语法正确、缩进正确）？
- [ ] 面试回答是否提供 30 秒 / 2 分钟 / 深挖追问？
- [ ] 代码示例是否注释足够清晰、能解释关键点与坑？
- [ ] 是否避免了不确定的源码细节“装懂”？
