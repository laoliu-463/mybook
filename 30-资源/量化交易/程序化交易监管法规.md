# 程序化交易监管法规

## 核心定义

基于《证券市场程序化交易管理规定（试行）》及沪深北交易所《实施细则》，程序化交易需遵循"先报告、后交易"原则，高频交易接受差异化从严监管。^[8,9]

## 关键要点

### 1. 高频交易认定标准

| 指标 | 阈值 | 说明 |
|------|------|------|
| **瞬时申报速率** | ≥ 300笔/秒 | 单账户每秒申报+撤单合计^[1] |
| **日内委托总量** | ≥ 20,000笔/日 | 单账户全日申报+撤单合计^[1] |

**触及任一阈值即被认定为高频交易**，需接受：
- 额外报告义务
- 差异化收费
- 重点监控^[3,5]

### 2. 异常交易行为（必须规避）

| 行为类型 | 定义 | 监管意图 |
|----------|------|----------|
| **瞬时申报速率异常** | 极短时间内申报量过大 | 防止冲击交易所系统^[4,6] |
| **频繁瞬时撤单** | 申报后 <100ms 立即撤单 | 打击诱导性交易(Layering/Spoofing)^[4] |
| **频繁拉抬打压** | 多只股票小幅批量操作 | 防止消耗流动性引发波动^[4,6] |
| **短时间大额成交** | 同机构不同产品集中同向交易 | 防止加剧市场波动^[4,6] |

### 3. 报告义务

> [!important] 先报告、后交易
> 未按规定报告不得进行程序化交易^[8,9]

**基础报告内容：**
| 类别 | 内容 |
|------|------|
| 账户信息 | 基本信息、资金来源、规模（含杠杆）^[7,9] |
| 策略信息 | 交易策略类型、逻辑简述^[7] |
| 技术信息 | 软件名称、版本、开发商^[7] |

**高频交易额外报告：**
- 服务器所在地
- 系统测试报告
- 系统故障应急方案^[5,10]

**北向资金（沪深股通）：**
按"内外资一致"原则，通过香港联交所报告^[9]

### 4. 合规风控机制要求

| 机制 | 要求 |
|------|------|
| **全链条风控** | 券商需对客户进行事前、事中、事后全流程管理^[9,13] |
| **系统故障应急** | 必须具备应对程序错误（死循环、算法失控）的处置能力^[6] |
| **合规核查** | 策略逻辑不得包含恶意诱导、虚假申报等违规意图^[8] |

### 5. 违规处罚措施（阶梯式）

| 级别 | 措施 |
|------|------|
| **轻微** | 口头警示、书面警示^[1,14] |
| **一般** | 暂停账户交易、限制交易^[1,14] |
| **严重** | 暂停主机托管(Co-location)，切断速度优势^[1,14] |
| **重大** | 通报证监会、现场检查、依法查处^[1,14] |

## 系统必须内置的合规功能

### 流量控制器 (Rate Limiter)

```
算法：令牌桶 (Token Bucket) 或 漏桶
精度：毫秒级监控
阈值：
  - 每秒报撤单 < 300笔
  - 全日报撤单 < 20,000笔
```
防止因程序Bug意外触发高频交易阈值^[2,7]

### 一键断路器 (Kill Switch)

```
功能：
  1. 立即物理阻断所有新订单发送
  2. 尝试撤销未成交订单

触发条件：
  - 系统逻辑异常（死循环重复报单）
  - 触及监管阈值
```
^[7,15]

### 撤单时间卫士

```
功能：强制设置订单最短停留时间
规则：限制 100ms 内禁止撤单
目的：防止触发"频繁瞬时撤单"认定
```
^[7]

### 杠杆率实时计算

```
功能：实时计算账户实际杠杆水平
目的：确保与报备的资金规模一致
      防止违规超额配资
```
^[7]

## 面试考点

**Q: 高频交易的认定标准是什么？**

> 单账户每秒申报/撤单 ≥ 300笔，或全日申报/撤单 ≥ 20,000笔。触及任一阈值即被认定，面临差异化收费和重点监控。

**Q: 量化系统必须内置哪些合规功能？**

> 四大核心功能：
> 1. 流量控制器（令牌桶算法限速）
> 2. 一键断路器（紧急停止）
> 3. 撤单时间卫士（防瞬时撤单）
> 4. 杠杆率实时计算（资金合规）
