## 核心概念

- 速率是传输速快慢，单位 bps。
- 带宽是链路能力上限，吞吐量是实际可达。
- 时延由传输、传播、处理、排队组成。
- RTT 是往返时间，常用于网络可达性评估。
- 时延带宽积衡量“管道里能放多少比特”。
- 利用率过高会放大排队时延。

## 源码/机制复盘（文字流程）

1) 速率和带宽决定单包在链路上的发送速度上限。
2) 吞吐量受端到端最慢链路或服务端能力限制。
3) 传输时延与包大小、带宽成正比。
4) 传播时延与距离和介质传播速率相关。
5) 排队时延随拥塞升高而急剧增加。
6) RTT 体现路径往返的综合时间成本。
7) 时延带宽积决定链路的“在途数据量”。

## 对比表

| 指标 | 关注问题 | 主要决定因素 |
| --- | --- | --- |
| 速率 | 传得快不快 | 设备能力 |
| 带宽 | 路修得宽不宽 | 运营商/链路 |
| 吞吐量 | 实际跑了多少 | 负载与瓶颈 |
| 时延 | 路上花多久 | 距离/拥塞 |
| RTT | 来回要多久 | 物理距离 |

## 可运行 Java 示例

`java
public class NetworkMetricsDemo {
    static double downloadTimeSeconds(double sizeBytes, double bandwidthBps) {
        // 统一单位：bytes -> bits
        return (sizeBytes * 8) / bandwidthBps;
    }

    static double bdpBits(double bandwidthBps, double rttSeconds) {
        // 时延带宽积决定在途数据量
        return bandwidthBps * rttSeconds;
    }

    public static void main(String[] args) {
        double bandwidth = 100_000_000; // 100 Mb/s
        double fileSize = 100 * 1024 * 1024; // 100 MB
        double rtt = 0.05; // 50 ms

        System.out.printf("Download time: %.2f s%n", downloadTimeSeconds(fileSize, bandwidth));
        System.out.printf("BDP: %.2f Mb%n", bdpBits(bandwidth, rtt) / 1_000_000);
    }
}
`

## 面试专栏

### ⚠️ 坑点/误区
- 把带宽当成实际下载速度。
- 忽略传播时延对长距离链路的影响。
- 只谈时延不考虑排队抖动。

### 追问链路
- 时延带宽积在工程上有什么意义？
- 为什么吞吐量常小于带宽？
- 短板效应如何影响端到端速度？
- 排队时延为何在高利用率下暴涨？
- RTT 与 TCP 拥塞控制有什么关系？

## 一分钟背诵版

1. 速率是每秒可传多少比特，单位 bps。
2. 带宽是链路能力上限，吞吐量是实际可达值。
3. 吞吐量被最慢链路或服务器限制。
4. 时延由传输、传播、处理、排队四部分组成。
5. 传播时延取决于距离和介质传播速度。
6. 排队时延最不稳定，受拥塞影响最大。
7. RTT 表示往返时间，常用 ping 测量。
8. 时延带宽积描述链路中在途数据量。
9. 提高带宽只能降低传输时延，无法降低传播时延。
10. 利用率过高会引发时延急剧上升。
