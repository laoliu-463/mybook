# HTTP - 缓存机制是如何工作的

## 一句话结论
HTTP 缓存通过存储资源的副本，减少服务器负载和客户端等待时间，提升 Web 性能。主要分为私有缓存（浏览器）和共享缓存（代理服务器）。

## 标准回答
- **私有缓存**：浏览器缓存，仅针对单个用户
- **共享缓存**：代理服务器缓存，多个用户共享
- **Cache-Control**：最重要的缓存控制头
- **ETag**：实体标签，用于条件请求验证
- **Last-Modified**：最后修改时间

## 为什么
每次请求都从服务器获取相同资源会造成：
1. 网络带宽浪费
2. 服务器负载增加
3. 用户等待时间变长

缓存机制通过复用本地/代理服务器的资源副本，解决上述问题。

## 对比
| 维度 | 私有缓存 | 共享缓存 |
|---|---|---|
| 位置 | 浏览器 | 代理服务器 |
| 作用域 | 单用户 | 多用户 |
| 典型场景 | 浏览器历史记录 | CDN、网关 |

## 缓存控制头详解
```http
# 缓存有效期3600秒，公开缓存
Cache-Control: max-age=3600, public

# 每次使用前必须验证
Cache-Control: no-cache

# 不缓存任何内容
Cache-Control: no-store
```

## 缓存策略流程
1. **新鲜度检测**：检查缓存是否过期（max-age）
2. **再验证**：发送条件请求（If-None-Match/If-Modified-Since）
3. **返回缓存**：如果有效则返回缓存副本

## 易错点
- `no-cache` 不是不缓存，而是每次使用前要验证
- `no-store` 才是真正的不缓存
- 私密资源用 `private`，公开资源用 `public`
- ETag 优先级高于 Last-Modified
