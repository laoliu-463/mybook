# Redis - 常见的缓存策略有哪些

## 一句话结论
Redis 常用缓存策略包括 Cache-Aside、Write-Through、Write-Behind，各有优劣。Cache-Aside 是最常用的模式。

## 标准回答
- **Cache-Aside**：查询时缓存，命中则返回，未命中查库后写入缓存
- **Write-Through**：写入时同步更新缓存和数据库，保证一致性
- **Write-Behind**：异步写入，性能最高但可能丢数据
- **过期策略**：TTL、LRU/LFU、主动刷新

## 为什么
直接访问数据库在高并发下会成为瓶颈，缓存通过将热点数据存入内存，大幅降低延迟并减轻数据库压力。不同的业务场景需要不同的缓存策略。

## 对比
| 策略 | 一致性 | 性能 | 复杂度 |
|---|---|---|---|
| Cache-Aside | 一般 | 高 | 低 |
| Write-Through | 高 | 低 | 中 |
| Write-Behind | 低 | 最高 | 高 |

## 缓存模式详解

### 1. Cache-Aside（旁路缓存）
```
应用 → 查缓存 → 命中 → 返回
       ↓ 未命中
     查数据库 → 写入缓存 → 返回
```

### 2. Write-Through（写穿透）
```
写入请求 → 同步写缓存 → 同步写数据库 → 返回
```

### 3. Write-Behind（异步写回）
```
写入请求 → 写缓存 → 返回 → 异步写数据库
```

## Redis 过期策略
- **TTL**：设置键的过期时间
- **LRU**：最近最少使用淘汰
- **LFU**：最不经常使用淘汰
- **主动刷新**：定期刷新热点数据

## 应用场景
- 会话存储（Session）
- 热点数据缓存
- 限流计数器
- 分布式锁

## 易错点
- Cache-Aside 在并发下可能出现缓存击穿（大量请求同时查库）
- Write-Behind 要做好数据丢失的补偿措施
- 缓存和数据库的一致性是永远的难题
- 设置 TTL 时要结合业务考虑数据更新频率
