# MySQL - 笔记管理系统数据库设计

## 一句话结论
笔记管理系统数据库通常包含用户表、笔记表、标签表及关联表，采用 InnoDB 存储引擎，支持软删除和数据隔离。

## 标准回答
- **核心表**：用户表、笔记表、标签表、用户标签关联表
- **存储引擎**：InnoDB（支持事务和外键）
- **字符集**：utf8mb4（支持 emoji）
- **特性**：软删除、数据隔离、级联删除

## 为什么
笔记管理系统需要：
- 多用户支持（用户表）
- 笔记 CRUD（笔记表）
- 标签分类（标签表 + 关联表）
- 数据恢复（软删除）

## ER 图
```
用户表 (users)
  ├── id (PK)
  ├── username
  ├── password
  └── deleted_at

笔记表 (notes)
  ├── id (PK)
  ├── user_id (FK)
  ├── title
  ├── content
  └── deleted_at

标签表 (tags)
  ├── id (PK)
  └── name

笔记标签关联表 (note_tags)
  ├── note_id (FK)
  └── tag_id (FK)
```

## 核心表结构
```sql
-- 用户表
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL UNIQUE,
    password VARCHAR(255) NOT NULL,
    deleted_at DATETIME DEFAULT NULL,
    INDEX idx_username (username),
    INDEX idx_deleted (deleted_at)
);

-- 笔记表
CREATE TABLE notes (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    title VARCHAR(255) NOT NULL,
    content TEXT,
    deleted_at DATETIME DEFAULT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    INDEX idx_user (user_id),
    INDEX idx_deleted (deleted_at)
);
```

## 设计要点
- 使用软删除（deleted_at）而非物理删除
- 为常用查询字段建索引
- 外键级联保证数据一致性
- 关联表实现多对多关系

## 易错点
- 不用自增 ID 用 UUID 会有性能问题
- 忘记加索引导致查询慢
- 软删除后查询要过滤 deleted_at
- 字符集要用 utf8mb4 不是 utf8
